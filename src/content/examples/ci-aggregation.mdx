Documentation goes stale the moment someone merges a change — and most teams notice only when an outdated answer surfaces in chat or a model hallucinates against a six-week-old README.

## The scenario

Teams that publish documentation — or feed it to an AI assistant — usually share one problem: their canonical sources live in many repositories, and the published view drifts the moment something merges. Manual syncs work for a week and then quietly fall behind.

## How this example handles it

This recipe moves the sync into CI. A scheduled GitHub Actions workflow (plus a `workflow_dispatch` trigger for ad-hoc rebuilds) checks out the citadel definition, runs `maester sync`, and uploads the resulting tree as a build artifact. The aggregation runs against the latest commit on every source repo, every time the workflow fires.

## What it gets you

- **Always-fresh artifacts.** Docs sites, RAG indexes, and downstream pipelines see the freshest content without anyone remembering to push a button.
- **Fully declarative, fully audited.** Every rebuild is a CI run with logs, diffs, and a permalink — drift is easy to spot and rollbacks are trivial.
- **Plays well with what you already ship.** Slots cleanly next to whatever you publish from Actions: Pages, S3, a docs bucket, an embeddings job.

> The pattern generalizes: any CI provider that can run `npx` on a cron schedule can drive a Maester sync the same way. GitHub Actions is just the most convenient default.
