Maester is a Node CLI. You install it once, then use it to manage two
kinds of configuration files: a `citadel.yaml` that lists the sources
you pull from, and (optionally) a `maester.yaml` on each source that
declares what it publishes.

## Install

```bash
npm install -g baller-maester
```

If you'd rather not install globally, every command below also works
through `npx`:

```bash
npx baller-maester
```

`npx baller-maester` is also the **top-level entry point** — running it with
no subcommand opens an interactive menu that picks the right next step
based on whether the current repo is a citadel, a maester, both, or
neither.

## Register your first citadel

Run the interactive walk-through from the root of the repository you
want to use as an aggregation point:

```bash
maester init
```

It will:

1. Prompt for a `baseDir` — the parent folder under which every
   default-destination source lands (pre-filled with `citadel`).
2. Prompt for one or more **source repositories** to register, each
   with a name, git URL, and ref.
3. Per source, offer an optional `includes` list — leave it blank to
   defer to the remote's own `maester.yaml`, or declare paths/globs
   directly when the remote has no manifest (a third-party repo, say).
4. Validate any `auth.type` env-var names so secrets never land in
   the `citadel.yaml`.
5. Add an idempotent `maester:sync` script to `package.json` if one
   exists.
6. Offer a final, recommended-by-default opt-in to install the
   [Grand Maester](/docs/concepts/grand-maester) agent skill — see
   below.

When it's done, you'll have a `citadel.yaml` at the repo root. Commit
it.

### About the Grand Maester step

The last prompt asks:

```text
Install the Grand Maester agent skill? (Y/n)
```

The Grand Maester is an optional integration that teaches your AI
coding agent (Claude Code, Codex CLI, Cursor, or any agent that reads
project-level `AGENTS.md`) where the citadel lives, which files are
[`canon`](/docs/concepts/document-state) vs `draft`, and to run
`maester status` before substantial reads so it never answers from a
stale corpus.

Accepting moves into target-agent selection (you can pick more than
one). Declining is fine — you can run `maester skill install` later.
Either way, init completes normally.

## Sync the citadel

To pull every source in your `citadel.yaml` into the local tree:

```bash
maester sync
```

Or to pull a single named source:

```bash
maester sync source-context
```

Sync uses partial clones and sparse checkouts — it only fetches the
paths the source actually publishes. Each source's destination
receives a small provenance marker recording the remote URL, ref,
resolved commit SHA, and active filter set.

## Check what's behind

Before relying on the citadel — or before an agent reads from it —
ask whether anything is behind:

```bash
maester status
```

Status is read-only. It uses the provenance markers to compare each
configured source against its remote and prints `up-to-date`,
`behind`, or `failed` for each. The exit code (`0` clean, `1`
behind, `2` failed) makes it easy to call from CI or from an agent's
pre-read hook.

## Configure a source (optional)

If you also want to declare what *this* repo publishes to other
citadels, run:

```bash
npx baller-maester
```

…and pick **"Configure this repo as a maester"** from the menu. The
walkthrough writes a `maester.yaml` at the repo root. No scripts are
scaffolded — the maester is purely a published manifest.
